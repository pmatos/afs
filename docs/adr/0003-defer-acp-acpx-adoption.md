# Defer Adoption of ACP / `acpx` for Agent Communication

AFS continues to speak Pi's JSONL JSON-RPC directly over child stdio (per ADR-0002) rather than reaching Pi through the Agent Client Protocol (ACP) or the [`acpx`](https://github.com/openclaw/acpx) CLI client. We chose this because ACP would replace a wire format AFS has just stabilized (PRD #2, sub-issues #43–#48), and going through `acpx`'s `pi-acp` adapter adds two extra process hops between AFS and Pi — the same kind of silent-drift surface that produced issue #40 — while `acpx` itself is explicitly alpha with breaking changes promised and would introduce a Node/npm runtime dependency under AFS's Rust binary. We also rely on `afs_reply`'s TypeBox `terminate: true` contract to make structured output a Pi-enforced invariant on every turn; ACP's general `tool_call` / `done` event stream is not a demonstrated drop-in for that finality guarantee.

Revisit when (a) `acpx` exits alpha and the ACP spec coverage roadmap stabilizes, (b) an ACP envelope can be shown end-to-end to preserve the `afs_reply terminate:true` structured-output guarantee, or (c) a successor PRD requires AFS to drive non-Pi agents (Codex, Claude Code, Gemini, …) — at which point a portable protocol becomes a feature requirement rather than a refactor.
